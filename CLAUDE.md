## What this is

zStellar is a shielded-payments dApp on Stellar (Soroban) Testnet. A user shields a public asset into a pool, transfers value privately inside the pool, and withdraws back to a public address — all without revealing amounts or the sender→receiver link on-chain. Zero-knowledge proofs (Groth16 over BN254 with Poseidon2) are generated client-side in WebAssembly; private transfers/withdrawals are submitted by a server-side relayer so the note owner's address never appears on-chain. It builds on Nethermind's Stellar Private Payments PoC. Testnet only, unaudited, never use with real funds.

Read `README.md` (root) for the full product/architecture narrative and the deployed contract addresses.

## Commands (run from `frontend/`)

This project uses **pnpm** (not npm) and **Biome** (not ESLint/Prettier).

```bash
pnpm install
pnpm dev                       # next dev (http://localhost:3000)
pnpm build                     # next build
pnpm lint                      # biome check
pnpm format                    # biome format --write
node scripts/setup-relayer.mjs # generate + Friendbot-fund the relayer, write .env.local
```

There is no test suite. Biome enforces 2-space indentation and organizes imports; run `pnpm lint` before considering a change done.

### Environment (`frontend/.env.local`, gitignored)

- `RELAYER_SECRET` — server-only (no `NEXT_PUBLIC_` prefix); read only by `/api/relay`. Generated by `setup-relayer.mjs`.
- `NEXT_PUBLIC_RELAYER_ADDRESS` — relayer's public address; absence disables relay (see `relayerConfigured()`).
- `NEXT_PUBLIC_STELLAR_RPC_URL`, `NEXT_PUBLIC_STELLAR_HORIZON_URL` — default to Stellar testnet endpoints in `src/lib/stellar/config.ts`.

## Architecture

The whole app is one page (`src/app/(main)/page.tsx` → `components/pages/(main)/MainPage.tsx`). `ActionPanel.tsx` is the core UI (Shield / Private Transfer / Private Withdraw + the proof stepper). Three layers matter:

### 1. The engine (`src/engine/`) — the boundary to the ZK/WASM core

`engine/index.ts` is the typed public API the UI calls: `shield`, `transfer`, `withdraw`, `getShieldedBalance`, `getMyShieldedAddress`, `depositWithAutoRegister`. It wraps a vendored WASM `WebClient` loaded at runtime from the static `/engine/js/web.js` bundle (`vendor/wasm-facade.js`) — that bundle is **not** webpack-bundled (note `webpackIgnore`/`@vite-ignore`). The `vendor/*.js` files are adapted from the Nethermind PoC; treat them as a vendored boundary and prefer changing `index.ts` over them.

Key behaviors encoded in `index.ts`:
- **Relayer routing** (`relaySetup`): when `useRelay` is set and a relayer is configured, the relayer becomes both the Soroban tx source and the `sender` arg of `transact`, and the proven tx is POSTed to `/api/relay`. Shield always submits via the user's wallet (the token pull needs user auth); transfer/withdraw default to the relayer.
- **OPFS reset on pool change** (`maybeResetStorage`): if the stored pool contract id differs from `CONTRACTS.pool`, it wipes OPFS and ASP localStorage flags. So changing `CONTRACTS.pool` invalidates every user's local note store by design.
- **ASP auto-register** (`depositWithAutoRegister` / `ensureAspRegistered`): first deposit registers the user in the ASP membership Merkle tree via `insert_leaf`, then retries the deposit while the chain syncs.
- Amounts are bigint **stroops** (7 decimals); use `xlmToStroops` / `stroopsToXlm`, never floats.

### 2. Stellar glue (`src/lib/stellar/`)

`config.ts` holds the network config and all deployed **contract addresses** (`CONTRACTS`) — the single source of truth, mirrored in the README table. `client.ts` reads balances / Merkle root / Friendbot funding. `register.ts` builds + submits `insert_leaf`.

### 3. Server routes (`src/app/api/`)

- `relay/route.ts` (Node runtime) — the privacy-critical path. Loads `RELAYER_SECRET`, signs the `sender.require_auth()` auth entry (`authorizeEntry`) and the tx envelope, submits via Soroban RPC, polls for confirmation. Only ever receives an already-proven, WASM-prepared tx from the browser.
- `rpc/route.ts` — a thin retrying proxy to the upstream Soroban RPC.

### Wallet (`src/features/wallet/`)

Freighter integration: `WalletProvider` + `useWallet` hold connection state; keys are **derived from a single Freighter signature** (`engine/vendor/wallet.js`) and cached in OPFS. `ReceiveButton` surfaces the shielded address (`noteKey:encKey`).

## Cross-origin isolation (do not break)

`next.config.ts` sets COOP `same-origin` + COEP `require-corp` globally. These are **required** for `SharedArrayBuffer` and OPFS, which the WASM prover and its Web Workers depend on. Any new cross-origin resource must be CORP-compatible or it will fail to load. The `/engine/*` assets are served `no-store` so a redeploy never serves a stale WASM bundle.
